# How / What / Why / Why-not 実践状況の調査

調査ブランチ: `investigate/how-what-why-comment-practices`  
対象: `main` 先端（調査開始時点）  
評価軸（依頼どおり）:

| 置き場 | 書くべきこと |
|---|---|
| コード本体 | **how**（実装そのもの） |
| テストコード | **what**（何を検証するか） |
| コミットログ | **why**（なぜ変えたか） |
| コードコメント（インライン） | **why not**（なぜ別案を採らないか／やってはいけないこと・非自明な制約） |
| ただしインライン | **一文**は why not 寄り。**一眼見てわからない複雑な処理**は how 説明可（複数行推奨） |
| ファイル説明 | **what で OK・本番では必須**（Go の package コメント / FE の `@fileoverview`） |

**why not** の意味: 「何をしているか」ではなく、「一見正しそうな別実装をなぜ採らないか」「フォールバックしない理由」「再登録しない理由」など、コードからは読み取りにくい否定・回避の根拠。

**複雑な処理の説明:** 同期の削除計画→確認→ダウンロードの順序、CAS 無しでの HEAD 再確認、トリガ SQL の文分割など、コードを追わないと意図が分からない箇所は手順・保証範囲の説明を書いてよい。自明な次行の復唱は不要。

**ファイル説明の例外:** package コメントと `@fileoverview` はファイル役割の what 説明であり、why not ルールの対象外。代わりに本番 `.go` / `.ts` / `.tsx` では必須（テスト・生成物は除外可）。記号ごとの godoc 網羅は必須にしない。

機械分類はヒューリスティックなので、件数は傾向指標。代表例で判断を補う。

---

## 総評

| 軸 | 実践度 | 一言 |
|---|---|---|
| テスト = what | **高い** | Go の関数名・Vitest の `it(...)` とも振る舞い記述が中心 |
| コミット = why | **高い** | `AGENTS.md` の方針どおり、本文に動機が多い（直近80件で本文 why シグナル約 74%） |
| インライン = **why not** | **低い（局所的には高い）** | sync/hotkey 等の複数行には why not がある。一文・JSX 見出しの大半は what |
| ファイル説明（package / `@fileoverview`） | **必須化した** | what 説明として許可。本番ファイルへ徹底（本調査ブランチで欠落分を追加） |
| コード = how | **該当** | 実装が how。問題はインラインが what/how を復唱していること |

結論: **テストとコミットは方針に沿う。インラインの正規は why not（全体では未徹底）。package / `@fileoverview` は what 説明として本番必須とし、欠落を埋めた。**

---

## 1. テストコード = what

### Go

- 命名は振る舞い指向が多い（例: `PushAbortsWhenRemoteHeadChangesMidUpload`, `PreservesClearedAt...WhenInputNil`, `IgnoresSymlinks`）。
- 直近集計: テスト関数 176 件のうち、When/Returns/Rejects/Preserves 等の振る舞い語を含むものが約 57%。残りは `TestParseHotkeyCombo` のような表駆動の総称名だが、中身はケース表で what を表現。
- 関数直前の複数行コメントも「〜することを確認する／検証する」と **what** を明示（テストとしては妥当）。

良い例:

```text
// TestBuildSaveTreeFollowsSymlinkedRoot は saveFolderPath 自体が ... でも
// 配下の通常ファイルがちゃんとスナップショットに入ることを確認する。
// これが崩れると Push が空の SaveSnapshot をアップロードして ...
```

（末尾に why も足しており、テスト doc としては過剰ではなく有用。）

### Frontend (Vitest)

- `it("Status returns push_needed with savesDiffer=false → pendingUpload は null のまま")` のように **観測可能な what** がタイトルになっている。
- 一部は「正常に動作する」など曖昧だが、全体として what 志向。

**判定: 実践されている。**

---

## 2. コミットログ = why

`AGENTS.md` 明示:

> 本文には「なぜそう変えたか」（動機・トレードオフ）を書く。コードから読める「何をしたか」は最小限

直近 non-merge 80 件の傾向:

- ほぼすべてに本文あり（Co-authored 行を除いても薄い/無しはごく少数）。
- 本文に「ため / 防ぐ / 原因 / 問題 / 届かない…」等の why シグナル: **約 59/80**。
- サブジェクトは Conventional Commits + 日本語の「何をしたか」要約が中心（これはリポジトリ規約どおり）。why は主に本文。

良い例（サブジェクト=要約、本文=why）:

```text
fix(screenshot): 起動同期でのホットキー多重登録を防ぐ
Startup 後の boot sync が notify/hotkey 更新を並列に再登録し、
Windows の already registered を誘発していた。...
```

```text
fix(sync): アップロード並列度の設定を ContentSyncService に反映
config は値コピーのため app.Config だけの更新では Push/Pull に届かなかった。
```

弱い例の傾向: サブジェクトが変更一覧の羅列で、本文も「何を直したか」の箇条書きが長いコミット（監査一括修正など）。動機は冒頭や各 bullet に散在するが、純粋な why 一文より what 比率が上がる。

**判定: 概ね実践されている（規約と実履歴が一致）。**

## 3. コードコメント = why not（再確認）

### 3.1 基準の訂正

初回調査ではコメントを「why（動機）」と読んだが、依頼の正規は **why not**。
その後の方針更新: **package コメント / `@fileoverview` は what 説明で OK かつ本番必須**（記号ごとの godoc 網羅は必須にしない）。

| 種別 | 例 | 判定 |
|---|---|---|
| why not | 「フォアグラウンドへフォールバック**しない**」「再登録**せず**に切り替える」 | インラインの正規 |
| soft why | 「〜するため TrimSpace する」（別案否定が弱い） | 近いが弱い |
| what インライン | 「ゲームデータを取得」 | インライン方針としては外れ |
| package / `@fileoverview` | 「このファイルは CAS 同期を提供する」 | **what で OK・必須** |

### 3.2 機械分類の傾向（本番コード）

Go（`internal/` + `main.go`、テスト除外）おおよそ:

| 分類 | 件数感 | 備考 |
|---|---|---|
| why not | 少（数十） | 複数行に偏る（single はごく少数） |
| soft why | 同程度 | 動機はあるが「何をしないか」が弱い |
| what | **多数**（数百） | godoc 一文の `〜を返す/提供する` が大半 |
| other | 多数 | 区切り線・型の補足など |

Frontend も同様: why not は数十、what/JSX セクション見出しが圧倒的に多い。

### 3.3 良い why not（実践されている箇所）

```text
// 対象ゲームがある場合は PID 必須。フォアグラウンドへのフォールバックはしない。
```

```text
// SetNotify はトースト通知の有効/無効を、OS ホットキーを再登録せずに切り替える。
```

```text
// Startup 済みの同一コンボを boot sync が再登録しようとして
// already registered になるのを防ぐ。
```

```text
// S3 に CAS が無いため完全な排他はできないが、...窓を大幅に縮小する
// （force 時はユーザーが上書きを選択済みのため省略）。
```

```text
// Wails webview 内で fragment/相対リンクを BrowserOpenURL に流すと不正な遷移になるため、
// http/https 以外は無視する。呼び出し側は e.preventDefault() をしないことで既定動作に任せる。
```

```text
// conflict を pull 確認に流すとローカルを黙って上書きしてしまう。
```

これらは「やらないこと／採らない別案」が明示されており、**why not として合格**。

### 3.4 why not になっていない例（ギャップ）

```text
// セーブファイルをアップロード（既存ブロブ一括確認 + 差分並列アップロード）
// ゲームデータを取得
// まずvisibleGamesから検索
// カスタムフック
```

```text
{/* 上段：ゲーム情報カード */}
{/* モーダル */}
```

```text
// OkResult は成功時の ApiResult を生成する。
```

（記号ごとの godoc。必須ではないが what 説明として許容。ファイル先頭の package コメントとは別。）

### 3.5 一文 vs 複数行

- **一文で why not になっている例は存在する**が少数（F12 非対応、PID フォールバックしない、等）。
- **良い why not の本体は複数行**に寄っている → 「複数行なら how 説明 OK」という例外の使い方としては妥当。
- 問題は、why not が不要な箇所に what 一文が残っていること、および why not であるべき非自明制約がコメント無しの箇所もあり得ること（本調査は有コメント側の品質が中心）。

**判定: why not は「危険な同期・ホットキー・パス・競合」周りでは実践されているが、リポジトリ全体のコメント方針としては未達。**

---

## 4. コード本体 = how

コードが how を担う前提は満たされている。  
ギャップは「コメントがコードの how/what を短く復唱」し、**why not が書いていない**点。

---

## 5. 層ごとのギャップまとめ

1. **最も方針に沿っている**: コミット本文の why、テストの what。
2. **インラインの正規 why not**: 一部の複雑ドメインでは実践。全体では what が主（今後の改善対象）。
3. **package / `@fileoverview`**: what 説明として許可し、本番必須。欠落は本ブランチで埋めた。
4. **一文 what / JSX 見出し**: インライン why not 基準ではノイズ。削除か、本当に別案否定があるときだけ残す。

---

## 6. 改善するなら（調査範囲外・提案のみ）

1. インラインは **why not**。複数行なら how 可。package / `@fileoverview` は what・必須（AGENTS.md に明記済み）。
2. 一文の「〜を取得/更新」は削除候補。
3. 既存の良い why not（hotkey 再登録しない、PID フォールバックしない、CAS 無し、など）をスタイル見本にする。
4. 新規変更では「なぜこの実装にしたか」より「なぜ X をしないか」を優先して書く。

このドキュメントは監査メモ。ファイル説明の徹底は本ブランチで実施済み。
